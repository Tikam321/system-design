# 🛡️ Web Security Quick Reference: CORS vs CSRF

This document serves as a quick reference guide to understand the core differences, working mechanisms, and prevention strategies for **CSRF** and **CORS**.

---

## 1. CSRF (Cross-Site Request Forgery)

### 📌 What it is
* It is a malicious **attack** that forces an authenticated user to execute unwanted actions on a trusted web application.
* **The Root Cause:** Browsers inherently practice **Ambient Authentication**—meaning they automatically attach saved session cookies to any request made to that specific destination domain, regardless of where the request originated.

### ⚙️ How it Works
1. A user logs into `bank.com` and a valid session cookie is stored in their browser.
2. Without logging out, the user visits a malicious website (`malicious.com`) in another tab.
3. The malicious site executes a hidden background script or form targeting `://bank.com`.
4. The browser automatically appends the `bank.com` session cookie to this request.
5. The bank server sees a valid session cookie, assumes the authorized user initiated it, and processes the transaction.

### 🛠️ Prevention Strategies
* **`SameSite=Lax` (Best & Default Practice):** Restricts the browser from sending cookies along with cross-site background requests (like scripts or forms). However, it allows cookies when a user explicitly clicks a direct link to navigate to your site.
* **`SameSite=Strict`:** Prevents the browser from sending cookies on *any* cross-site request, including standard link clicks from third-party sites (causes users to appear logged out when navigating from external links).
* **Anti-CSRF Tokens:** A unique, unpredictable, and secret one-time token generated by the server for each user session. This token is embedded inside hidden form fields or custom headers. Since the malicious site cannot access the token, the server rejects unauthorized requests.
* **Authorization Headers:** Avoid keeping session identifiers or JWTs in cookies. Instead, store them in `LocalStorage` or `SessionStorage` and pass them manually via custom headers (e.g., `Authorization: Bearer <token>`). Browsers never attach these headers automatically.

## 2. CORS (Cross-Origin Resource Sharing)

### 📌 What it is
* It is **not an attack**; it is a **browser security mechanism**.
* By default, browsers enforce the **SOP (Same-Origin Policy)**, which strictly prevents scripts on one website from reading sensitive data from another website.
* CORS is a way for a server to safely relax this policy and instruct the browser to allow cross-origin requests for trusted domains (e.g., allowing frontend `localhost:3000` to fetch data from backend `://mysite.com`).

### ⚙️ How it Works
1. A frontend application hosted on **Origin A** sends an API request to a backend server on **Origin B**.
2. For sensitive or non-trivial requests, the browser first sends an automatic `OPTIONS` request (known as a **Preflight request**) to check permissions.
3. The backend server must respond with specific headers, such as `Access-Control-Allow-Origin: Origin A`.
4. If the header matches or allows the origin, the browser permits the frontend to read the data. If it doesn't match, the browser blocks the response and throws a **CORS Error**.

### 🛠️ Configuration Best Practices
* **Allow Specific Domains:** Explicitly define trusted origins on the backend server: `Access-Control-Allow-Origin: https://myfrontend.com`.
* **Avoid Wildcards (`*`):** Never use `Access-Control-Allow-Origin: *` in production environments, as it allows any external site to read your API data.
* **Credential Sharing:** If your CORS requests require session cookies or authorization headers to pass through, the backend must also set `Access-Control-Allow-Credentials: true`.

## 🔄 Quick Summary Matrix

| Feature | CSRF | CORS |
| :--- | :--- | :--- |
| **Nature** | A malicious **attack** exploiting browser cookie behavior. | A browser-enforced **security protocol** protecting resources. |
| **Objective** | To execute unauthorized state-changing actions on a server. | To regulate which external domains can read API responses. |
| **Who Suffers?** | The user (unwanted actions) and the server (compromised data). | The frontend developer (dealing with blocked API responses during development). |
| **Who Fixes It?** | Backend developers (by implementing SameSite cookies or tokens). | Backend developers (by configuring the correct response headers). |
